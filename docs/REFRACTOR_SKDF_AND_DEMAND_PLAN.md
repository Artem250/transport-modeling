# План упрощения проекта: СКДФ, спрос, маршруты и расчётные модули

## 1. Главная проблема текущего состояния

Код содержит полезные части, но они смешаны:

- модель данных;
- загрузка/сохранение JSON;
- назначение спроса;
- статический расчёт V/C и LOS;
- визуализация;
- экспериментальные идеи для гидродинамики/CTM.

Из-за этого программа выглядит сложнее, чем сама методика. Для ВКР лучше сделать меньше функций, но чтобы было понятно:

1. какие данные берутся из СКДФ;
2. какие параметры рассчитываются;
3. какие параметры являются допущениями;
4. каким методом получен каждый результат.

## 2. Что считать базовым источником данных

Пользователь хочет по возможности использовать только данные СКДФ. Доступные поля сегмента:

| Поле | Смысл в модели |
|---|---|
| segment_object_id | идентификатор сегмента |
| road_name_segment | название дороги |
| road_part_id_segment | идентификатор участка/части дороги |
| start_km_segment | начало сегмента по километражу |
| finish_km_segment | конец сегмента по километражу |
| traffic_segment | интенсивность / спрос на сегменте |
| capacity_segment | пропускная способность сегмента |
| top_speed_segment | максимальная/разрешённая скорость |
| lanes_segment | число полос |
| roadway_width_segment | ширина проезжей части |

Эти поля достаточны для статического анализа сегментов:

- длина сегмента;
- интенсивность;
- пропускная способность;
- V/C;
- LOS по V/C;
- грубая оценка скорости и задержки.

Но этих данных недостаточно для полноценной модели перекрёстков и маршрутов без допущений.

## 3. Чего нет в СКДФ-сегментах

Из перечисленных полей нельзя напрямую получить:

- геометрию линии на карте, если нет координат;
- направление движения;
- входящие и исходящие связи в узлах;
- поворотные доли: налево / прямо / направо;
- фазы светофора;
- зелёное время;
- длительность цикла;
- потоки насыщения по группам полос;
- OD-матрицу;
- реальный выбор маршрута водителями;
- состав потока по типам автомобилей;
- почасовой профиль интенсивности, если traffic_segment является средним или агрегированным значением.

Поэтому все такие параметры должны храниться как `assumption`, `manual`, `default`, `derived`, а не выглядеть как данные СКДФ.

## 4. Предлагаемый принцип

Не запрещать допущения полностью, а явно маркировать источник каждого параметра.

Пример:

```json
{
  "capacity_pcu_h": 1800,
  "source": "skdf.capacity_segment",
  "confidence": "observed"
}
```

или:

```json
{
  "green_sec": 30,
  "source": "default_assumption",
  "confidence": "assumed",
  "note": "Нет данных о фазах светофора в СКДФ"
}
```

Так работа будет честной: СКДФ используется максимально, а недостающие параметры явно обозначены как допущения.

## 5. Как сейчас устроен спрос

В текущем коде есть два режима:

1. `routes` — пользователь заранее задаёт маршрут и спрос на него.
2. `route_split_coefficients` — задаётся входной поток в граничный узел и коэффициенты распределения по маршрутам.

Это не полноценная транспортная модель спроса и не динамическое назначение потоков. Это сценарное распределение потоков по заранее заданным путям.

Для ВКР это нормально, если честно формулировать:

> В работе не решается задача построения OD-матрицы и динамического выбора маршрутов. Используется сценарное задание спроса: либо через заранее заданные маршруты, либо через коэффициенты распределения входных потоков.

## 6. Что нужно упростить первым

Первый этап рефакторинга должен быть маленьким и безопасным:

1. Не менять визуализацию.
2. Не удалять CSV/JSON/OSM.
3. Не мержить старую гидродинамическую ветку целиком.
4. Ввести явные названия для текущих расчётов:
   - `StaticVCAnalysisService` вместо общего `AnalysisService`;
   - `ScenarioDemandAssignmentService` или оставить `DemandAssignmentService`, но задокументировать ограничения.
5. Добавить слой нормализации СКДФ-сегментов:
   - `skdf_segment_id`;
   - `length_km` из `finish_km - start_km`;
   - `traffic_pcu_h` или `traffic_veh_h`;
   - `capacity_pcu_h`;
   - `speed_limit_kph`;
   - `lanes`;
   - `roadway_width_m`;
   - `data_source`.
6. Добавить простую проверку: какие поля взяты из СКДФ, а какие являются допущениями.

## 7. Минимальная честная модель для ближайшего этапа

Ближайшая цель:

> СКДФ-ориентированная статическая модель сегментов с явной маркировкой источников данных и сценарным распределением спроса по заранее заданным маршрутам.

Это лучше, чем сразу пытаться сделать полноценную LWR/CTM-модель.

После этого можно добавлять CTM как отдельный динамический модуль.

## 8. Что делать с гидродинамикой позже

Гидродинамику/CTM стоит подключать только после чистки данных:

1. сначала нормальные links из СКДФ;
2. затем movements на узлах;
3. затем signals/phases, если они есть или заданы как допущения;
4. затем CTM cells;
5. затем динамические результаты: density, queue, travel_time, spillback.

CTM не должен подменять статический расчёт. Это отдельный расчётный движок поверх той же сети.
