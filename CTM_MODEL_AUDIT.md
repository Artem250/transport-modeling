# Аудит CTM-модели для ВКР

Дата: 2026-05-19  
Основная ветка аудита: `гидравлика_в_codex`.

Этот документ фиксирует, что в текущей CTM-ветке уже можно считать расчётной моделью, что можно защищать в ВКР, а что лучше честно описать как ограничение. Код этим коммитом не менялся.

## 1. Важно про ветки

В `main` полноценного CTM-ядра я не увидел: там основной расчёт ближе к статическому HCM/LOS-анализу и визуализации результатов. CTM-код найден в ветках `гидравлика_в_codex`, `codex/гидравлика` и `дипломная-работа-без-идей-92424`.

Для ВКР разумнее фиксировать `гидравлика_в_codex` как расчётную CTM-ветку, потому что там есть связка:

- `models.py` — структуры `Project`, `Network`, `Link`, `Source`, `Sink`, `Movement`, `SimulationConfig`;
- `network_migration.py` — восстановление sources/sinks/movements из топологии;
- `dynamic_analysis.py` — построение runtime-сети и запуск симуляции;
- `network_dynamic.py` — CTM-ячейки, storage, sending, receiving, скорость, очередь;
- `ctm.py` — шаги симуляции и расчёт потоков;
- `traffic_viz.py` — визуализация уже агрегированных результатов.

## 2. Что сейчас реально моделируется

Текущая динамическая часть — это макроскопическая CTM-модель на направленной сети.

Схема такая:

```text
Project -> Network -> Link -> DynamicLink -> Cell -> CTMSimulator -> link.results -> traffic_viz
```

Каждый link делится на CTM-ячейки. В ячейке хранится `occupancy_pcu`, то есть количество приведённых автомобилей. Для link задаются длина, число полос, свободная скорость, скорость обратной волны, jam density и capacity.

Базовые формулы, которые можно защищать:

```text
storage = jam_density_per_lane * lanes * cell_length_km
capacity_step = capacity_pcu_h * dt / 3600
sending_i = min(occupancy_i, capacity_step)
receiving_i = min(storage_i - occupancy_i, wave_limit_step, capacity_step)
flow_i_to_i+1 = min(sending_i, receiving_i+1)
```

Это нормальная CTM-логика.

## 3. Как устроены узлы

Узлы сейчас не являются полноценной физической моделью перекрёстка. Они представлены через `Movement`:

```text
from_link -> to_link, split_ratio, capacity, control
```

В `ctm.py` поток через узел ограничивается:

- отправляющей способностью последней ячейки входящего link;
- принимающей способностью первой ячейки выходящего link;
- split ratio;
- capacity конкретного движения;
- control: uncontrolled / signalized / roundabout / priority.

Это можно защищать как агрегированную модель узла, но не как полноценную модель конфликтных точек, полос и очередей по направлениям.

## 4. Что уже защищаемо

Защищаемо:

- links дискретизируются на ячейки;
- поток между ячейками считается через sending/receiving;
- есть источники и стоки;
- есть движения через узлы;
- есть динамические показатели: плотность, скорость, очередь, задержка, throughput;
- можно сравнивать базовый сценарий и сценарий с локальным снижением capacity.

Хорошая формулировка для ВКР:

> Реализован программный прототип макроскопической динамической модели дорожной сети на основе Cell Transmission Model для сценарной оценки влияния локального снижения пропускной способности участка дороги.

## 5. Главные ограничения

1. **CTM-ветка не совпадает с `main`.** Перед защитой нужно явно указать, какая ветка и какие файлы являются расчётной моделью.

2. **Перекрёстки агрегированные.** В модели есть movements и control, но нет конфликтных точек, отдельных поворотных полос и полноценного фазового расчёта.

3. **Sources/sinks/movements часто эвристические.** `network_migration.py` достраивает их по топологии. Это удобно, но не равно реальным OD-данным.

4. **Split ratios не являются полноценным route choice.** Динамическая корректировка поворотных долей — это эвристика, а не traffic assignment.

5. **Нужна явная проверка сохранения потока.** В обновлении используется `max(..., 0.0)`, поэтому для защиты лучше иметь тест/отчёт с `conservation_error`.

6. **`traffic_viz.py` не доказывает динамику.** Он показывает агрегированные `link.results`, а не покадровое состояние CTM.

## 6. Что делать дальше, не раздувая код

Безопасные задачи:

1. Зафиксировать `гидравлика_в_codex` как рабочую CTM-ветку.
2. Добавить 2-3 маленьких теста: один link, два links подряд, один вход и два выхода.
3. Проверять `occupancy >= 0`, `occupancy <= storage`, `flow <= capacity_step`, `conservation_error`.
4. Подготовить один сценарий: базовая сеть и сеть со сниженной capacity на выбранном link.
5. Сделать таблицу сравнения: `avg_density`, `avg_speed_kph`, `max_queue_pcu`, `queue_length_m`, `Delay_sec`, `throughput_pcu`.

## 7. Стоп-правило

Не надо сейчас добавлять OD-матрицы, новую архитектуру перекрёстков, полноценный route choice, микромодель полос и новый GUI. Это снова приведёт к росту кода быстрее, чем растёт понимание.

Текущая модель уже может быть основой ВКР, если её защищать как CTM-прототип с эвристическими граничными условиями, а не как реалистичный цифровой двойник города.
